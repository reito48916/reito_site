# 命名規則の0埋めと接尾語より接頭語優先は義務教育で日本標準にしてほしい

例えば

- 1.txt
- 72.txt
- 573.txt

という3つのファイル。これを一番プリミティブなABC順にすると、72よりも573が先に来てしまう。ファイル名の長さも不定だ。

親切に72を先にしてくれるシステムも世の中にはあふれているため、こうなるとファイルを利用する側のシステムによってこれらを使った機能の出力結果が変わってしまう事も多々ある。

ここで、

- 001.txt
- 072.txt
- 573.txt

という感じで0埋めをすると、その揺れを無くすことが出来る。課題点はしいて言えば、あらかじめ何桁まで通し番号が存在しうるのかが分かっていないといけない事だが…、それが問題にならない場合は、こうしておくことにデメリットはないだろう。

###

もう一点、こちらはなかなか意識しても難しい事ではあるのだが、共通の語句は後ろではなく前に持ってきた方が皆嬉しい、という話だ。例えば

- CommonICar.txt
- SpecialCar.txt
- UltimateCar.txt

よりも、

- CarCommon.txt
- CarSpecial.txt
- CarUltimate.txt

の方が、名前順にしたときにそろってくれて助かる、ということ。英文的には修飾語が前に来るので、何も考えないと自然と共通語が後ろに来てしまう事はあるあるだ。

ただし、上の例だとケースによってはCarではない色々なものについてCommonのファイルがあって、むしろCarよりもCommonやSpecialの方が共通語だ、ということもあり得る事には注意が必要だ。つまりは初期状態ではどの語が共通語となるか判別がまだ出来ない場合もあり、結果として接尾語が共通語になってしまっていても責められないケースも多い。

とはいえ、ファイル名やプログラミングの識別子命名で特にそういった事情無くこれらの規則に則っていないサンプルを見ると、少し情けない気持ちになる。

余談だが、ゲーム開発だと運営型のゲームで3桁を想定していたものが運営長期化で4桁にたどり着いてしまう、という事があり、それについては個人的には結構誉高いことなんじゃないかな、と思う。別にその時999個ある既存のモノに0を付け足すことはそこまで苦労する事では無いし…。
